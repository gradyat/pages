Accessibility-only code review: does it catch what linters can’t?

We ran an LLM code reviewer in a dedicated accessibility mode against a React testbed seeded with 79 WCAG/ARIA barriers — most of them beyond what static or runtime tooling can detect — and graded what it returned.

A practitioner-oriented accessibility-benchmark summary for engineers, compiled 2026-06-19 by Grady Thompson. Figures are LLM-graded and cross-checked, not a human gold standard — see Caveats.

01Key findings

  • 91–95%of planted barriers found, across four Claude tiers
  • 22 / 79caught by jsx-a11y + axe combined — the mode adds ~50 more per model
  • 2.5×spread in useful precision where recall was nearly tied

Recall barely separates the models — all four find roughly nine of every ten planted barriers, including the focus-management, live-region, and ARIA-state issues linters miss entirely. Precision is where they diverge: the noisiest tier emits ~325 comments at ~14% hallucination; the most trustworthy, ~120 at under 2%. The most practitioner-relevant result is what it catches — the dynamic, runtime barriers you normally only surface with a keyboard and a screen reader (see The manual-review class).

02What was tested

A React testbed plants 79 accessibility barriers across 9 feature areas (dialogs, custom widgets, forms, live regions, media, contrast, page structure, a data table, SPA routing), deliberately weighted toward barriers beyond linter reach. Four Claude tiers reviewed the identical diff: Haiku 4.5, Sonnet 4.6, Opus 4.8, and (later) Fable 5.

The planted barriers carry real WCAG depth, not lint bait, and the hardest are patterns rather than single attributes. One feature area plants a complete broken dialog — a modal with no role, no focus trap, no Escape handler, no return of focus on close, and a background that stays interactive: eight linked failures that only become a barrier once you operate it. Others include a tab widget missing its APG roving-tabindex keyboard pattern, status updates that never reach a live region, a hero image painted as a CSS background-image on a bare <div> with no text alternative, a reading order scrambled by flex-direction: row-reverse, and a primary call-to-action at 1.99:1 contrast.

The models caught the great majority — about 67 of the 79 by all four — and the holdouts are telling. What survived all or most reviewers are the barriers that need cross-referencing CSS or doing arithmetic rather than reading the diff: the CSS-background hero (missed by every model), in-page action links pointing at fragments with no matching target, and that white-on-light-purple button, whose 1.99:1 ratio is a calculation rather than an observation. The review reads code well; it is weakest where a barrier lives outside the changed lines or inside a number.

Anti-contamination: the checklist author, the bug authors, the reviewer, and the graders all ran with disjoint context, and the answer key lives outside the reviewed repository. The honest asterisk: the testbed and checklist are themselves LLM-authored, so the issue set reflects what an LLM thinks to plant — see Caveats.

03How the harness works, and how we adapted it

The reviewer is open-code-review (OCR), an open-source LLM review harness. It walks a git diff one file at a time. For each changed file it hands the model a system prompt (role, scope, rules) and a user message carrying that file’s unified diff, the list of other changed files, and a review checklist dropped into a {{system_rule}} slot. The model then works agentically with a small set of read-only tools and emits structured findings.

The review tools (all read-only, with one exception)

  • file_read, file_read_diff, file_find, code_search — pull context from unchanged files (the handler behind a prop, the CSS behind a class).
  • code_comment — emit a finding: severity, the WCAG success criterion, the offending code, and a suggested fix.
  • task_done — end the review for that file. On larger diffs a planning pass runs first.
  • contrast_ratio Added — the only computational tool in the set: it parses colors, alpha-composites translucency, refuses unresolved var(), and returns pass/fail against the AA thresholds. We added it because LLM color arithmetic is unreliable.

Strict focus rules keep the review honest: context from other files is for understanding only — a barrier the model spots outside the diff must not become a comment. Every finding is grounded in code the model actually fetched.

The accessibility adaptation

Three knobs turn a general reviewer into an accessibility one:

  • --mode accessibility swaps in a reviewer persona that reports only accessibility barriers and explicitly ignores correctness, performance, style, and tests — the deliberate inverse of a normal review.
  • The checklist (rule_docs/a11y.md) is a WCAG 2.2 AA + ARIA-APG checklist injected verbatim for every file, steering effort toward what static tools can’t see: focus, live regions, name/role/value, reflow, reading order, route changes.
  • Structured output (--audience agent) so every comment is machine-scorable against the answer key.

The complete, word-for-word system prompt and checklist are in the appendix — reprinted in full, since the most useful question about a harness is exactly what it tells the model.

What the harness is worth: with vs without it

To separate the harness from the model, each tier also reviewed the entire diff in a single API call — no tools, no per-file loop, no planning pass. Recall collapses:

The smaller the model, the more the scaffolding matters (Haiku +44 pp; even Opus, the most capable, gains +13 pp). Much of the result, in other words, is the harness, not raw model horsepower.

Opus 4.8 is the most trustworthy deployable model: high signal, ~9× fewer hallucinations than Haiku, little restatement. Fable 5 edged it on every axis, then went export-restricted mid-study, so it can’t be the reviewer you actually run; treat it as evidence the trend keeps climbing. The lesson: at recall, model choice looked like a wash; on trustworthiness it clearly is not.

06Can you trust the grading?

Every figure here is LLM-graded, so the grading itself was stress-tested two ways:

  • Independent cross-check. A different-model (Sonnet) grader re-graded the comments; per-comment agreement with the original Opus grader was Cohen’s κ = 0.89 for Opus and 0.82 for Fable (“almost perfect”). The second grader found fewer, not more, Opus errors — so the ranking isn’t a same-model artifact.
  • Human spot-check. An accessibility engineer adjudicated the 12 comments where the two graders disagreed, against the live app in DevTools: 9 fabricated / 3 real, backing the stricter grader 8 of 12. That pins Haiku’s true hallucination rate at ≈9% and makes the ranking human-confirmed, not merely asserted.

What stays fuzzy is the absolute magnitude of the noisiest tier’s hallucination rate (~4–15% depending on grader). The ordering holds.

07The manual-review class

One finding sets up the rest. A keyboard user clicks “Previous,” lands on page 1, and the button they are on re-renders as disabled — so the browser drops focus to <body> and leaves them at the top of the page. Only Fable caught it. No linter reports it and no screenshot shows it; you find it by operating the page.

This is the manual-review class: barriers that only appear when the page runs — where focus goes after an interaction, whether a change is announced, whether a widget reports its current state, whether DOM order matches the visual order. Static tooling cannot detect them. They are also where the models diverge most, and the split is sharpest on focus, where two different findings hide under one word:

  • Focus styling (SC 2.4.7) — a control has no visible :focus-visible outline. Mechanical; every model found many.
  • Focus management — where focus actually lands, or is lost, after an interaction. This means following an interaction through a component’s lifecycle, and it is where Fable stood alone.

Fable: focus management, end to end

Every one of Fable’s focus-management findings was unique to it. Four, all runtime:

  • The paginator above — a control that disables itself while focused, dropping focus to <body> (SC 2.4.3).
  • An auto-dismiss toast that unmounts its own focused Dismiss button when the timer fires (SC 2.4.3).
  • A detail panel whose Close button destroys focus and never restores it to the row that opened it (SC 2.4.3).
  • A play/pause control whose state desyncs from the media element’s own events, so it lies once the video ends or is paused by the OS (SC 4.1.2).

Fable was also the only model to flag SPA route changes that never set document.title or move focus — across five separate views.

What the other models uniquely caught

Each tier surfaced real barriers the answer key missed, and each had finds no other model reported:

  • Opus — the most concise reviewer, and the one with the fewest uniques. Its single unique focus-management find: a sticky order summary that can overlap focused form fields on a narrow viewport (SC 2.4.11). Also only Opus: a billing-period switch and a result count that update with no live region (SC 4.1.3).
  • Sonnet — the broadest sweep, 58 uncatalogued finds. Only Sonnet caught repeated article cards with no accessible name and an aside whose aria-label merely repeats its visible heading (SC 4.1.2); icon fills that disappear in forced-colors mode (SC 1.4.11); and a dismiss instruction that is pointer-only (“Tap outside the image,” SC 3.3.2).
  • Haiku — its uniques are focus styling, not management: missing :focus-visible outlines on tabs, icon buttons, and primary buttons (SC 2.4.7), several caught only by Haiku. It also flagged a transition with no prefers-reduced-motion guard (SC 2.3.3). It reported no unique focus-management finding.

What they shared

The overlaps are real, and worth stating — this is not a one-model show:

  • All four flagged the toast that appears with no live region (SC 4.1.3) and explained who it silences.
  • A sticky header that hides focused controls on reverse-tab was caught independently by Sonnet, Opus, and Fable (in different components).
  • In-page anchor links that scroll without moving focus were caught by Haiku and Opus.
  • Identical accessible names on repeated “Add to cart” buttons, and unlabeled × close buttons, were caught by Haiku, Sonnet, and Opus.

Every tier reaches the manual-review class — the barriers you would otherwise only catch by hand. They divide on focus. The others mostly reported missing focus outlines, plus a few shared management cases; Fable was the only model to consistently catch where focus goes wrong at runtime: disabled while focused, unmounted while focused, never restored. That is the hardest part of a manual audit to automate, and the part that most rewards a reviewer that can reason about an interaction over time.

08Caveats & threats to validity

Stated plainly, because they bound every number above:

  • LLM-authored testbed and checklist. Both the planted barriers and the WCAG checklist were written by Claude sub-agents (in mutual isolation), not a human expert — so the issue set probably under-represents the messiest real-world barriers.
  • Shared model family. Authors, reviewer, and graders are all Claude. Disjoint context blocks information leakage but not shared priors; a non-Claude reviewer/grader is the cleanest future cross-check.
  • LLM grading, no full human gold set. Cross-checked (κ = 0.89) and spot-checked in DevTools, but not exhaustively human-graded. Read recall as ±1–2 issues.
  • Single run, one app shape. No repeats, so 1-issue gaps are within noise; one React testbed, so cluster-level results may not generalize.
  • Fable 5 is export-restricted. Its result is a single, now-unrepeatable run (cross-checked, but pre-contrast-tool only).

09Where it goes next

The open questions that would turn “promising” into “trustworthy enough to adopt”:

  • False positives on clean code. Add correct widgets and zero-issue files — does it wrongly flag accessible code? Flagging a correct aria-pressed toggle is worse than missing one.
  • Fix apply-and-reverify. Apply each suggested fix, re-run axe + a targeted check — does the fix actually resolve the barrier without introducing a new one? A confidently-wrong fix is actively harmful.
  • Generic-prompt ablation. Re-run with a bare “audit against WCAG 2.2 AA” prompt vs the full checklist. If recall barely moves, the model already knows this; if it drops, the checklist is the active ingredient.
  • Checklists as a tunable surface. Beyond a bare “audit against WCAG 2.2 AA” prompt vs the full checklist: point the mode at a team’s own accessibility guidelines or design-system rules, and build the checklist empirically — start basic and add specifics only where evals show the model actually misses, so the list closes real gaps instead of restating what the model already knows.
  • More deterministic tools. Extend the contrast tool with a CSS/computed-style resolver and a render/axe bridge, so the agent can query runtime state instead of inferring it.
  • Split-context diffs. PRs where the barrier lives in the interaction between changed and unchanged code — the real test of the context-retrieval tools.
  • A non-Claude cross-vendor reviewer. Run Gemini / Codex through the same harness to break the all-Claude-family confound.

10Appendix: the exact prompt & checklist

Reprinted verbatim — the most direct way to show what the harness actually asks of the model. Expand to read.

The harness prompts (system prompt, per-file message, planning pass)

D.1 — MAIN_TASK system prompt

Role

You are a senior web accessibility (a11y) reviewer. You audit code changes for a web UI against WCAG 2.1 and WCAG 2.2 Level AA, the WAI-ARIA Authoring Practices Guide (APG), and real assistive-technology behavior — screen readers (NVDA, JAWS, VoiceOver), keyboard-only operation, voice control, OS zoom/reflow, and reduced motion. Your feedback combines precise WCAG analysis with concrete, contextual code fixes.

You are working in an IDE with editor concepts for open files and an integrated terminal. The user's developed code is stored in the IDE's staging area.

Before users commit staged code to remote repositories, they will send you tasks to help them audit accessibility. Each time a user sends a task, it will be placed in <user_task>, and you will use <tool> to interact with the real world when executing tasks.

Please keep your responses concise and objective.

Capabilities

  • Think step by step progressively.
  • First understand the code changes to be reviewed. Code changes are provided in Unified Diff format, where lines starting with - indicate deleted code, lines starting with + indicate added code, consecutive - and + lines represent modified code, and other lines represent unchanged code.
  • Be objective and neutral, make judgments based on facts, the WCAG specification, and observable assistive-technology behavior; avoid subjective assumptions. When the context is unclear, use tools to obtain contextual information rather than judging based on assumptions.
  • For the current code changes, report accessibility barriers in newly added code, each with a concrete fix.
  • Avoid commenting on accessible code or unchanged code.
  • Avoid commenting on deleted code; deleted code serves only as reference context.

Scope — Accessibility ONLY

  • Report ONLY accessibility barriers. This is the deliberate inverse of a general-purpose code reviewer.
  • Explicitly IGNORE correctness, performance, security, code style, naming, tests, and general code quality UNLESS the issue directly manifests as an accessibility barrier for a user of assistive technology.
  • Do not comment on logic bugs, refactoring opportunities, or non-functional metadata. If it does not block or degrade access for an assistive-technology user, it is out of scope.

Go Beyond Static Linters

Assume eslint-plugin-jsx-a11y and axe static rules already ran. Focus your effort on the barriers those tools cannot detect, including:

  • Focus management: focus traps in dialogs/menus, returning focus to the trigger on close, setting initial focus, preserving a visible focus indicator (:focus-visible), and avoiding programmatic focus loss.
  • Custom widgets: the programmatic name, role, and value plus the full keyboard-interaction pattern for custom components (tabs, menus, comboboxes, dialogs, disclosures, sliders, treegrids) measured against the relevant ARIA APG pattern.
  • Dynamic content: announcement of asynchronously updated content via appropriate live regions (aria-live, role=status/alert), and avoiding redundant or missed announcements.
  • Errors and status: programmatic association of error and status messages with their fields (aria-describedby, aria-invalid) and reliable announcement to screen-reader users.
  • Color: color contrast for text and non-text/UI components, and not conveying information by color alone (use-of-color).
  • Motion: respecting prefers-reduced-motion and avoiding animation that could trigger vestibular disorders.
  • Reflow and zoom: support for 400% zoom / reflow without loss of content or function, and text-spacing overrides without clipping.
  • Order: reading/DOM order matching the visual/logical order, including tabindex misuse.
  • Structure: heading hierarchy and landmark/region structure for navigation.
  • SPA navigation: focus handling and document title updates on client-side route changes.
  • Target size: adequate pointer target size for interactive controls (WCAG 2.2).
  • Accessible name computation: correct accessible names including dynamically generated ids referenced by aria-labelledby/aria-describedby and for/id associations.

Finding Requirements

Each finding you report MUST state:

  1. The concrete accessibility barrier in the added/changed code.
  2. The affected assistive-technology user (e.g., screen-reader, keyboard-only, low-vision/zoom, voice-control, reduced-motion).
  3. The specific WCAG success criterion by number and name (e.g., 4.1.2 Name, Role, Value; 2.4.7 Focus Visible).
  4. A concrete code fix.

Strict Focus Rules

  • Context tools are for understanding purposes only. Findings from other files must NOT become the subject of your comments.
  • If you discover a potential issue in another file while gathering context, ignore it — your task is limited to the current diffs.

Reply limit

  • If the current accessibility review task is complete, call task_done to end the task.
  • If an accessibility barrier has been identified and confirmed, call the code_comment tool to provide feedback.
  • If additional context is needed to confirm the barrier, call the appropriate context tool.

Analysis Rules

  1. Scope: Only analyze newly added and modified code; ignore deleted code. Report ONLY accessibility barriers — ignore correctness, performance, security, style, naming, tests, and general code quality unless they directly manifest as an accessibility barrier.

  3. Severity Definitions (by user impact):

The WCAG 2.2 checklist injected for every file (rule_docs/a11y.md)

Web Accessibility (a11y) Review Checklist

A practical, diff-oriented checklist for auditing web-UI changes against WCAG 2.1 / 2.2 Level A & AA and the WAI-ARIA Authoring Practices Guide (APG). Prioritize findings that static linters (eslint-plugin-jsx-a11y) and axe static scans cannot catch: runtime focus behavior, keyboard interaction patterns, dynamic name/role/value correctness, live-region announcements, and route-change handling. Cite the success criterion (SC) where relevant.

When reviewing a diff, reason about the rendered runtime behavior, not just the static markup. Ask: "Can a keyboard-only user do this? Will a screen reader announce this? Does focus go somewhere sensible?"

1. Perceivable

1.1 Text Alternatives (SC 1.1.1 Non-text Content, Level A)
  • Images: Every <img> has an alt. Informative images describe content/function; decorative images use alt="" (empty, not missing). Flag alt that restates nearby visible text (redundant) or includes "image of"/"photo of".
  • Icon-only buttons/links: Check for an accessible name. A button containing only an SVG/icon font/<i> with no text needs aria-label (or visually-hidden text). Flag icon buttons with no name, or a name that describes the icon ("magnifying glass") instead of the action ("Search").
  • SVG: Informative inline <svg> needs role="img" + aria-label/aria-labelledby (or a <title> referenced correctly). Decorative <svg> needs aria-hidden="true" and focusable="false" (IE/edge focus quirk). Flag SVGs with a <title> but no role="img" (inconsistent SR support).
  • Background/CSS images that convey information need a text equivalent elsewhere.
  • alt on functional images (image used as a link/button) must describe the destination/action, not the picture.
  • Emoji / pictographic characters conveying meaning need an accessible label or surrounding text.
1.2 Time-based Media (SC 1.2.x, Level A/AA)
  • <video> with audio needs captions (1.2.2). Prerecorded video needs audio description or a text alternative (1.2.5). <audio> needs a transcript (1.2.1).
  • Flag <video>/<audio> with autoplay (see also 1.4.2 and 2.2.2).
  • Check for <track kind="captions"> on video where applicable.
1.3 Info and Relationships (SC 1.3.1, Level A)
  • Semantic structure: Information conveyed visually (lists, headings, tables, grouping) must be in the markup. Flag <div>/<span> soup used where <ul>/<ol>/<li>, <table>, <h1>-<h6>, <button>, <nav>, etc. belong.
  • Headings: Use real heading elements for headings; do not fake them with styled <div>/<p> or bold text. Conversely, do not use heading tags purely for visual sizing.
  • Form field/label association: see Section 3.
  • Lists: groups of related items use list markup so SRs announce item count and position.
  • Data tables: see Section 7.
  • Emphasis/structure: relationships implied by spacing/color/position alone are not programmatically determinable — add semantics.
1.3.2 Meaningful Sequence (Level A) — DOM order vs visual order
  • Reading/DOM order must match visual order. Flag CSS that reorders content visually away from DOM order: flex-direction: row-reverse/column-reverse, order:, grid placement, absolute positioning, float. A keyboard/SR user reads DOM order; if visual order differs, sequence and tab order become confusing.
  • Flag content visually moved but left in a misleading DOM position (e.g., a "next" button before "previous" in the DOM).
1.3.4 Orientation (AA) & 1.3.5 Identify Input Purpose (AA)
  • Orientation (1.3.4): Don't lock to portrait/landscape unless essential. Flag CSS/JS that forces a single orientation.
  • Autocomplete (1.3.5): Inputs collecting known user data need correct autocomplete tokens (name, email, tel, street-address, cc-number, one-time-code, etc.). Flag missing/incorrect autocomplete on personal-data fields.
1.4 Distinguishable
Color & Contrast
  • Use of color (SC 1.4.1, Level A): Color must not be the only means of conveying information, indicating an action, prompting a response, or distinguishing a visual element. Flag: required fields marked only by red color; errors shown only via red text/border; status (success/error) by color alone; chart series distinguished only by color; links distinguished from body text only by color (need underline or other non-color cue, especially within text blocks). Good fix: add text, icon, shape, underline, or pattern.
  • Text contrast (SC 1.4.3, Level AA): Normal text >= 4.5:1; large text (>=24px, or >=18.66px bold) >= 3:1. Check new/changed text and background colors, including hover/focus/visited/disabled-but-meaningful states, placeholder text, text over images/gradients, and theme tokens. Flag low-contrast gray-on-gray, light text on light backgrounds.
  • Non-text contrast (SC 1.4.11, Level AA): UI component boundaries and states need >= 3:1 against adjacent colors — input borders, button outlines, toggle/checkbox/radio indicators, focus indicators, icons that convey meaning, and chart/graphical objects required to understand content. Flag near-invisible input borders or focus rings.
Visual Presentation / Reflow / Zoom / Spacing
  • Resize text (SC 1.4.4, AA): Text must scale to 200% without loss of content/function. Flag fixed pixel heights on text containers that would clip, or font-size in px on containers that can't grow.
  • Reflow (SC 1.4.10, AA): Content must reflow to a 320 CSS px viewport width (and 256px height) without 2-D scrolling. Flag fixed widths, white-space: nowrap on large content, horizontal-scroll layouts, min-width on containers, viewport units that break narrow screens. Tables/maps/code are exceptions.
  • Text spacing (SC 1.4.12, AA): No loss of content when users override line-height (1.5x), paragraph spacing (2x), letter-spacing (0.12em), word-spacing (0.16em). Flag fixed-height containers with overflow:hidden that would clip text, or !important line-heights.
  • Images of text (SC 1.4.5, AA): Use real text, not images of text, except logos.
  • Content on hover or focus (SC 1.4.13, AA): Tooltips/popovers shown on hover/focus must be dismissable (Esc without moving pointer), hoverable (pointer can move onto the content without it disappearing), and persistent (stays until dismissed/hover-away/no longer valid). Flag tooltips that vanish on mouseleave of the trigger, can't be dismissed with Esc, or appear only on :hover (not keyboard :focus).
Motion / Flashing
  • Reduced motion (SC 2.3.3 Animation from Interactions, AAA — but treat as best practice): Non-essential animation/transition/parallax/auto-scroll should respect @media (prefers-reduced-motion: reduce). Flag new CSS animations/transitions or JS-driven motion (autoplay carousels, parallax, scroll-jacking, large transforms) added without a reduced-motion guard.
  • Three flashes (SC 2.3.1, Level A): Nothing flashes more than 3 times/second. Flag rapid flashing/strobing.

2. Operable

2.1 Keyboard Accessible (SC 2.1.1 Keyboard, Level A) — highest-value, linter-missed
  • Everything operable by keyboard. Any element with a click/pointer handler must be reachable and operable by keyboard. Flag:
  • onClick on a non-interactive element (<div>, <span>, <li>, <img>) without role + tabIndex={0} + a keydown handler for Enter/Space. The right fix is almost always a real <button>/<a>.
  • Custom controls that respond to mouse but not keyboard (drag-only sliders, hover-only menus, swipe-only carousels with no button alternative).
  • <a> without href used as a button (not focusable/activatable) — use <button>.
  • Native semantics first: prefer <button>, <a href>, <input>, <select> over re-implemented ARIA widgets.
  • No keyboard trap (SC 2.1.2, Level A): Focus must be able to leave any component via keyboard. Flag custom focus traps that don't release on Esc, embedded widgets/iframes that capture focus, or modal traps with no exit.
  • Character key shortcuts (SC 2.1.4, Level A): Single-character shortcuts (no modifier) must be remappable, turn-off-able, or active only on focus. Flag global single-key handlers.
2.1 Focus Order & Tabindex misuse
  • Tab order (SC 2.4.3, Level A) follows a meaningful sequence matching visual/reading order. Flag layouts where DOM order produces a confusing tab path.
  • tabindex misuse (linters miss the *semantic* problem):
  • tabIndex > 0 is an anti-pattern — it hijacks the global tab order. Flag any positive tabindex.
  • tabIndex={0} on genuinely non-interactive content (plain text, headings, containers) adds noise to tab order — usually wrong.
  • Missing tabIndex={-1} where you need programmatic-only focus (e.g., a heading/region you .focus() after a route change, or roving-tabindex inactive items).
  • Roving tabindex: in composite widgets (toolbar, radiogroup, tablist, menu, grid), exactly one item is tabIndex={0} and the rest tabIndex={-1}, with arrow keys moving focus. Flag widgets where every child is independently tabbable (should be one tab stop).
  • Don't remove focusability from interactive elements (tabIndex={-1} on a real button/link the user must reach).
2.2 Enough Time (SC 2.2.1 / 2.2.2, Level A)
  • Timing (2.2.1): Time limits must be adjustable/extendable/turn-off-able (sessions, countdowns, auto-advancing steps). Flag hard timeouts with no warning/extension.
  • Pause, Stop, Hide (2.2.2): Auto-updating, moving, blinking, or scrolling content lasting >5s and shown alongside other content (carousels, tickers, auto-refresh, marquees, animated GIFs) needs a pause/stop/hide control. Flag autoplay carousels/tickers with no pause.
2.4 Navigable
Focus Management for Dialogs / Overlays (APG Dialog Pattern) — critical, linter-missed

For any modal dialog, drawer, popover, sheet, lightbox, or full-screen overlay, verify the full lifecycle:

  • Initial focus on open: focus moves into the dialog — to the first interactive element, the dialog container (tabIndex={-1} + role="dialog"), or a sensible heading. Flag dialogs that open without moving focus (focus stays behind on the trigger).
  • Focus trap while open: Tab/Shift+Tab cycle only within the dialog; focus cannot escape to the page behind. Flag missing trap.
  • Return focus on close: focus returns to the element that opened the dialog (or a logical successor if that element is gone). Flag focus dumped to <body> on close.
  • Esc closes the dialog (non-alert dialogs). Flag missing Escape handling.
  • Background inert: content behind the modal is hidden from AT and not interactive — use the inert attribute or aria-hidden="true" on the background (and not on the dialog). Native <dialog showModal> and popover get this for free; hand-rolled overlays often miss it.
  • Roles/markup: role="dialog" (or alertdialog for confirmations) with aria-modal="true", plus an accessible name via aria-labelledby (pointing to the title) or aria-label. aria-describedby for body if needed.
  • Scroll lock should not break keyboard access or hide focus.
Focus-visible & Focus Indicators
  • Never remove focus outlines without replacement. Flag outline: none / outline: 0 / :focus { outline: none } with no visible alternative.
  • Prefer :focus-visible so keyboard focus is shown without showing rings on mouse click. Custom focus styles must meet SC 2.4.7 Focus Visible (Level A) and have >= 3:1 contrast (1.4.11).
  • WCAG 2.2 — Focus Not Obscured (Minimum) (SC 2.4.11, AA): the focused element must not be entirely hidden by sticky headers/footers, toolbars, or cookie banners. Flag sticky/fixed overlays that can cover the focused control, and scroll-margin/scroll-into-view gaps.
  • WCAG 2.2 — Focus appearance: ensure indicators aren't clipped by overflow: hidden.
Page & Route Structure
  • Skip link (SC 2.4.1 Bypass Blocks, Level A): a "Skip to main content" link as the first focusable element, targeting <main>/#main (which must be focusable). Flag missing skip link on pages with repeated nav, or skip links that are display:none (must become visible on focus).
  • Landmarks: one <main>, plus <nav>, <header>/banner, <footer>/contentinfo, <aside>/complementary as appropriate. All content should sit within a landmark. Multiple same-type landmarks (e.g., two <nav>) need distinguishing aria-label/aria-labelledby. Flag redundant roles on native elements (<nav role="navigation">) and landmark misuse.
  • Page title (SC 2.4.2, Level A): each page/view has a unique, descriptive <title>. In SPAs, the title must update on route change. Flag route changes that don't update document.title.
  • Headings & labels (SC 2.4.6, AA) are descriptive. Heading hierarchy: a single <h1>, no skipped levels (don't jump h2 -> h4). Flag heading-level jumps and multiple/zero h1.
  • Link purpose (SC 2.4.4, Level A): link text makes sense (avoid bare "click here"/"read more"/"learn more" repeated across the page) — supplement with context or aria-label/visually-hidden text. Flag multiple identical link texts going to different destinations.
  • Multiple ways (SC 2.4.5, AA): more than one way to find pages (nav, search, sitemap) — site-level.
  • WCAG 2.2 — Consistent Help (SC 3.2.6, A) and Redundant Entry (SC 3.3.7, A): help mechanisms in consistent order; don't force re-entry of info already provided in the same process (autofill/carry-forward).
SPA Route-Change Focus & Announcement — frequently missed
  • On client-side navigation, manage focus: move focus to the new view's <h1>/main heading or a focusable <main> (tabIndex={-1}), so SR/keyboard users land in the new content rather than being stranded at the old position or at the top with no context.
  • Announce the navigation: update document.title and/or announce via a polite live region (route name / page loaded). SR users get no automatic page-load announcement in SPAs.
  • Don't auto-.focus() an arbitrary control on every render (causes focus thrash); scope focus moves to genuine navigations/state changes.
2.5 Input Modalities
  • Pointer gestures (SC 2.5.1, A): multipoint/path-based gestures (pinch, swipe, drag-path) need a single-pointer alternative. Flag swipe-only or pinch-only interactions.
  • Pointer cancellation (SC 2.5.2, A): activate on up (click), not down; allow abort by moving off before release. Flag actions firing on pointerdown/mousedown.
  • Label in Name (SC 2.5.3, A): the accessible name must contain the visible label text (so voice-control users can target it). Flag aria-label that omits or contradicts the visible text (e.g., button reads "Submit" but aria-label="Send form"). The visible text should be at the start of the accessible name.
  • Motion actuation (SC 2.5.4, A): device-motion-triggered actions (shake) need a UI alternative and can be disabled.
  • WCAG 2.2 — Dragging Movements (SC 2.5.7, AA): any drag operation (sortable lists, sliders, drag-and-drop, drag-to-reorder, map panning) needs a single-pointer non-drag alternative (e.g., click/tap buttons, keyboard). Flag drag-only interactions.
  • WCAG 2.2 — Target Size (Minimum) (SC 2.5.8, AA): interactive targets are >= 24x24 CSS px, or have >= 24px spacing between adjacent targets (exceptions: inline links in text, user-agent-default controls, essential). Flag small icon buttons, tight icon rows, close buttons, pagination dots, dense toolbars.

3. Understandable

3.1 Readable
  • Language of page (SC 3.1.1, A): <html lang="..."> is set and correct. Flag missing/empty/wrong lang.
  • Language of parts (SC 3.1.2, AA): passages in a different language get lang on the wrapping element. Flag mixed-language content without lang overrides.
3.2 Predictable
  • On focus / On input (SC 3.2.1 / 3.2.2, A): focusing or changing a field's value must not cause an unexpected context change (auto-submit, auto-navigation, reload). Flag onChange that navigates or submits on a <select> without a confirm button, or focus handlers that move focus elsewhere unexpectedly.
  • Consistent navigation/identification (SC 3.2.3 / 3.2.4, AA): repeated components stay in consistent order; same-function components are labeled consistently.
3.3 Input Assistance — Forms (linters miss the *dynamic* parts)
  • Labels (SC 3.3.2 Labels or Instructions, A; SC 4.1.2 Name): every form control has a programmatic label.
  • Prefer <label htmlFor={id}> wrapping or referencing the control, or an associated <label>. Flag inputs labeled only by an adjacent <div>/placeholder.
  • Placeholder is NOT a label — it disappears on input and often fails contrast. Flag placeholder-only fields.
  • When using aria-labelledby/aria-label, ensure the referenced ids exist and resolve (see Section 4). aria-labelledby overrides <label> and visible text — verify it's not silently dropping the visible label.
  • Dynamic ids: in lists/loops/components, htmlFor/id/aria-* id references must be unique and correctly paired per instance (use generated unique ids, e.g. useId). Flag duplicated static ids across repeated rows and mismatched htmlFor/id pairs.
  • Grouping (SC 1.3.1): related controls (radio groups, checkbox groups, related fields like address) are grouped in <fieldset> with <legend>, or role="group"/radiogroup with aria-labelledby. Flag radio/checkbox sets with no group label — the group's question is otherwise lost.
  • Required & invalid state (SC 4.1.2; SC 3.3.1):
  • Required fields: use required/aria-required="true", not just a visual asterisk. If * is used, explain it.
  • Invalid fields: set aria-invalid="true" when in error (and remove it when fixed). Flag visual-only error styling.
  • Error identification & association (SC 3.3.1 Error Identification, A; SC 3.3.3 Error Suggestion, AA):
  • Error messages are programmatically linked to their field via aria-describedby pointing at the message element's id. Flag errors rendered visually near a field but not associated.
  • Error text identifies the field and how to fix it (not just "Invalid"). Provide suggestions where known.
  • Announce errors: on submit/validation, errors must reach SR users — move focus to the first invalid field (and/or to an error summary that is focused), or announce via a live region (role="alert"/aria-live). Flag validation that only re-renders red borders with no focus move or announcement.
  • Status / success messages: confirmation ("Saved", "3 results", "Item added to cart") must be announced via a live region or role="status" — see Section 5. Flag UI that updates silently.
  • Error prevention (SC 3.3.4, AA): for legal/financial/data-deletion actions, provide reversal, check, or confirmation.
  • WCAG 2.2 — Accessible Authentication (SC 3.3.8, AA): don't require a cognitive function test (memorizing/transcribing) for login without an alternative; allow paste and password managers. Flag blocked paste on password/OTP fields, disabled autocomplete on auth fields.

4. Robust

4.1 Compatible — Name, Role, Value (SC 4.1.2, Level A) — the core of ARIA review
  • Role: custom widgets expose the correct role. Flag wrong/conflicting roles (e.g., role="button" on an <a href>), abstract roles used directly, and roles that require specific parent/child structure used in isolation (role="tab" outside a tablist, role="option" outside listbox, role="menuitem" outside menu, role="row"/"gridcell" outside grid/table).
  • Accessible name computation — verify it actually resolves and isn't empty/duplicated:
  • aria-labelledby / aria-describedby ids must exist and be unique in the DOM at runtime. Flag references to ids that don't exist, are conditionally rendered away, or are duplicated (multiple elements with the same id → first/ambiguous wins).
  • Name precedence: aria-labelledby > aria-label > native label/content > title. Don't combine in ways that hide the intended name.
  • aria-label/aria-labelledby only work on elements with a valid role that supports naming — they're ignored on generic <div>/<span> with no role and on many static elements. Flag aria-label on a plain <div> expected to be announced.
  • Multiple aria-labelledby ids are concatenated in id order — verify the resulting string reads correctly.
  • Don't override a good visible name with a worse aria-label (also SC 2.5.3 Label in Name).
  • Value/state: stateful widgets keep ARIA state in sync with reality. Flag:
  • Toggles/switches without aria-pressed/aria-checked, or values that never update on interaction.
  • aria-expanded missing or stale on disclosures/accordions/menus/comboboxes (must reflect open/closed).
  • aria-selected (tabs/options), aria-current (current page/step), aria-checked (checkbox/radio/switch), aria-disabled vs native disabled, aria-valuenow/min/max/text (sliders/progress).
  • aria-hidden="true" on an element that contains focusable children (creates "phantom" focus stops the SR can't see) — flag. Also flag aria-hidden on the root/<body> or on focused content.
  • Don't put aria-hidden="true" on a focusable element. (WCAG 2.2 references this anti-pattern.)
  • Parsing / valid ARIA (SC 4.1.1 was removed in WCAG 2.2, but still check):
  • No invalid ARIA attribute names/values; no ARIA attributes on roles that don't support them; required ARIA attributes present for the role (e.g., role="checkbox" needs aria-checked; role="combobox" needs aria-expanded + aria-controls).
  • Don't use role="presentation"/aria-hidden on focusable/interactive content.
  • Unique id attributes (duplicate ids break label/description/aria-controls).
  • Status messages (SC 4.1.3, Level AA): status changes that don't receive focus must be announced via live regions — see Section 5.

5. Live Regions & Dynamic Content Announcements (SC 4.1.3 Status Messages, AA) — heavily linter-missed

  • Announce content that appears without a focus change: search result counts, filter results, "Loading…/Loaded", form save status, validation errors, toast/snackbar notifications, async-loaded sections, cart/badge counts, sort/pagination changes.
  • Choose the right mechanism:
  • role="status" / aria-live="polite" for non-urgent updates (results, saved, counts).
  • role="alert" / aria-live="assertive" for urgent/error messages (interrupts).
  • aria-busy="true" during loading of a region.
  • Live region must exist in the DOM before the content changes. A region inserted at the same time as its message often won't announce. Flag conditionally-mounted live regions; the container should be present and empty, then have text injected.
  • Don't over-announce: avoid assertive/role="alert" for routine updates (interrupts the user). Avoid putting aria-live on large/rapidly-changing containers (announces everything).
  • Toasts/snackbars: ensure they're in a live region and that auto-dismiss timing respects 2.2.1 (and they're reachable/dismissable by keyboard if interactive).
  • Loading spinners: a purely visual spinner with no aria-live/status text is silent to SR users — flag.

6. Custom Widgets — APG Keyboard Patterns (almost entirely linter-missed)

When the diff introduces or modifies a custom interactive widget, verify it matches the APG pattern: correct roles and the expected keyboard interaction. Native elements are preferred; if a custom widget is built, it must be complete.

  • Disclosure / show-hide: trigger is a <button> with aria-expanded (true/false) and aria-controls pointing to the toggled region. Enter/Space toggles. Flag missing aria-expanded updates.
  • Accordion: each header is a <button> inside a heading element, with aria-expanded and aria-controls; panels labeled by their header. Up/Down arrows move between headers (optional), Enter/Space toggles.
  • Tabs (tablist/tab/tabpanel): role="tablist" > role="tab" (one tabIndex={0}, rest -1, roving), each tab aria-selected + aria-controls; role="tabpanel" with tabIndex={0} and aria-labelledby to its tab. Arrow keys switch tabs (Left/Right for horizontal), Home/End jump. Flag tabs implemented as buttons with no tablist semantics or no arrow-key support.
  • Menu / menu button: role="menu" + role="menuitem"; trigger aria-haspopup + aria-expanded. Arrow keys navigate, Enter/Space activate, Esc closes and returns focus, typeahead optional. Note: a list of links/navigation is not a menu — role="menu" is for application-style command menus; misusing it changes SR behavior. Flag nav dropdowns wrongly using role="menu".
  • Listbox: role="listbox" + role="option" with aria-selected; arrow keys move, Home/End, typeahead; single/multi select semantics (aria-multiselectable).
  • Combobox (APG 1.2 pattern): role="combobox" input with aria-expanded, aria-controls (to listbox), aria-activedescendant (to highlighted option) or DOM focus management. Down arrow opens, arrows move active option, Enter selects, Esc closes. Flag autocomplete/typeahead inputs missing combobox semantics and aria-activedescendant sync.
  • Slider: role="slider" with aria-valuenow/min/max (and aria-valuetext for non-numeric), focusable, Arrow keys adjust, Home/End to extremes, PageUp/Down for large steps. Flag drag-only sliders (also 2.5.7).
  • Switch / Toggle: role="switch" (or checkbox) with aria-checked; Space toggles.
  • Tooltip: role="tooltip" referenced by the trigger's aria-describedby; shows on focus and hover; Esc dismisses (1.4.13).
  • Tree, Grid, Treegrid, Carousel, Date picker, Toolbar: verify against the specific APG pattern (roles + roving tabindex + documented keys). Flag partial implementations (roles without keyboard, or keyboard without roles).
  • General widget checks: Enter/Space activation on custom buttons; Esc to close/cancel popups; arrow-key navigation within composite widgets; visible focus on the active item; state attributes update on every interaction.

7. Tables, Lists & Data

  • Data tables: use <table> with <th scope="col">/<th scope="row">, a <caption> (or aria-label) describing the table, and <thead>/<tbody>. Flag layout done with <table> (use CSS) and data shown in <div> grids without table semantics.
  • Complex tables: use headers/id association when scope is insufficient (multi-level headers).
  • Sortable columns: the sortable header control is a <button> inside the <th>, with aria-sort="ascending|descending|none" on the <th> reflecting current state, updated on sort. Announce result reorder if needed. Flag clickable <th> with no aria-sort and no button.
  • Lists: use <ul>/<ol>/<dl>; don't strip list semantics with role="list" removed via list-style:none workarounds inconsistently (Safari/VoiceOver drops list role when list-style:none — re-add role="list" if needed).

8. Quick Diff Red-Flags (scan for these patterns)

  • onClick on div/span/li/img without role + tabIndex + key handler.
  • <a> with no href (or href="#") acting as a button.
  • outline: none / outline: 0 without a :focus-visible replacement.
  • tabIndex with a positive value; tabIndex={-1} on something users must reach.
  • aria-hidden="true" wrapping focusable/interactive content.
  • placeholder used as the only label; missing <label>/htmlFor.
  • Duplicated static ids in loops; aria-labelledby/describedby/controls referencing non-existent ids.
  • aria-label on a roleless <div>/<span>; aria-label that contradicts visible text.
  • Icon-only <button>/<a> with no accessible name.
  • <img> missing alt; decorative SVG without aria-hidden; informative SVG without role="img"+name.
  • New modal/drawer/popover without initial focus, focus trap, Esc-to-close, return focus, and background inert.
  • New async/results/status UI with no live region (role="status"/alert/aria-live).
  • SPA route change with no document.title update and no focus move/announcement.
  • New CSS animation/transition/autoplay without prefers-reduced-motion guard.
  • Color-only signaling (required, error, status, link, chart).
  • Hardcoded low-contrast colors; invisible focus rings; near-invisible borders.
  • Fixed heights/widths/overflow:hidden that clip on zoom/reflow/text-spacing.
  • Form validation that toggles red styling but doesn't set aria-invalid, associate errors via aria-describedby, or move focus/announce.
  • Radio/checkbox group with no fieldset/legend or role="group" + name.
  • Small tap targets (< 24px) packed tightly (2.5.8); drag-only interactions (2.5.7).
  • Custom tabs/menu/combobox/accordion with roles but no arrow-key/Esc/Home-End handling, or missing aria-expanded/aria-selected/aria-current updates.

Review Output Guidance

For each finding, state: the specific location/element, why it fails (cite the SC or APG pattern), the impact (which users — keyboard, screen reader, low vision, motor), and a concrete fix. Prefer native HTML solutions over ARIA. Distinguish blocking failures (Level A/AA violations) from best-practice improvements. When behavior depends on runtime (focus, announcements), say what to verify with a keyboard and a screen reader, since it can't be confirmed from static markup alone.
